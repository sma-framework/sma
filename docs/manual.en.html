Manual · SMA V2

SMA: a memory and coordination framework for AI agents

What it is, how it works, which commands you have, and what happens on its own. In plain words, no jargon.

1. What it is, in a nutshell

You work with several AI assistant windows at once (each window is called a "terminal" here). This way of working has two chronic problems:

  1. The assistant forgets. Every new conversation starts from a blank slate. Rules, past mistakes, and agreements have to be repeated, or you hope it finds them on its own.
  2. Terminals get in each other's way. Two windows edit the same file, grab the same database number, and every so often that ends in breakage.

SMA solves both. It is a framework that gives assistants a permanent memory and teaches terminals to see each other. The second version adds a third thing: a learning loop. The assistant states in advance what its work will change, compares the promise against the fact, and every miss becomes a permanent rule.

2. How it is built: engine and overlay

Inside runs an open workflow engine derived from gsd-core (MIT license; provenance is tracked in the repo files). It knows how to drive a project step by step: discuss the task, write a plan, execute it, verify the result.

The SMA overlay adds what the engine cannot do:

The engine (workflows)The SMA overlay
Drives a project: discussion, plan, execution, verificationPermanent memory: project knowledge as notes, the assistant receives the relevant ones automatically
Keeps documents per phaseTerminal coordination: who is working, on what, warnings before a conflict
Knows nothing about several windows at onceShared counters: migration and release numbers are handed out in turn, no collisions
Executes the plan and stopsLearning loop: predictions, promise-vs-fact scoring, reflexes born from misses

The simplest way to think about it: the engine is the motor, SMA is the car built around it. Everything is deterministic scripts over files and git: no daemons, no databases, no network dependencies.

3. The foundation: memory

Your project's knowledge lives in the project folder as small notes, one fact per note: a rule, a lesson from a mistake, a decision, a work status. Every note carries tags from a strict vocabulary.

  • The core is always with the assistant. The most important rules and current blockers load into every conversation automatically, from the first second.
  • The rest on demand. Working on payments? The assistant pulls only the notes tagged "finance", not the whole memory at once.
  • The index is machine-built. The memory table of contents is generated by a script from the notes. A manual edit is flagged immediately by the checker.
  • There is a quality inspector. pnpm sma lint finds notes without tags, with vocabulary typos, with broken links, or gone stale.

4. The foundation: terminals see each other

Every window leaves a "pulse": a small file saying who it is and what it is working on. From that, several abilities follow:

  • Who is working right now. pnpm sma status shows all active windows and what each one is doing.
  • A claim on files. A terminal declares "I am taking these files" (pnpm sma claim). If another terminal reaches for the same files, it gets a warning BEFORE the edit, with the name of whoever holds them. Work is never blocked: a warning only.
  • Numbers without races. A migration or release number comes from pnpm sma next-slot: the first terminal gets 067, the second automatically gets 068.
  • The "push in progress" signal. While one terminal ships code to your deploy host, the others see it and stay out of the way.

5. V2 pillar: predictions

What it does. Every work plan states up front what will measurably change: a metric, a check command, a threshold. After execution, a deterministic scorer compares promise against fact and writes the verdict to the journal. A calibration ledger shows in which areas the system is wrong most often: those areas get stricter oversight.

The command: scoring runs at plan close (predict-score); results surface in the report.

6. V2 pillar: reflexes

What it does. A scored miss becomes a rule with a firing condition. Before each subsequent assistant action, the system checks whether the action matches a condition, and if it does, injects a warning straight into the conversation, BEFORE the edit. Like a child who touches boiling water once and never again. Noise controls are built in: repeats are muted, and every rule has a kill switch.

The command: fires on its own through a pre-action hook; the rule list and its statistics surface in the report.

7. V2 pillar: corpus health

What it does. To keep the memory sharp at hundreds of notes, the system tidies up regularly: merges duplicates, promotes lessons that keep proving true, marks stale material, and catches contradicting notes from parallel terminals before they reach the main branch.

The command: pnpm sma consolidate (scheduled cleanup) and pnpm sma lint (quality check).

8. V2 pillar: coordination with teeth

What it does. Checkable hard rules of the project graduate from "a request in prose" to automatic gates: the system itself catches forbidden actions before they run. File claims gain a second tier: for hot files, a soft deny can be enabled while another terminal holds a fresh claim. Every shared counter in the project is handed out in turn.

The command: works on its own through hooks; the picture is visible in pnpm sma status.

9. V2 pillar: a resilient harness

What it does. Every plan keeps a progress journal: if an executor dies mid-work, a new session resumes from the exact spot in minutes instead of reconstructing the picture from crumbs. A stall detector notices when the assistant is going in circles. Task waves are computed from dependencies and run in parallel when it is safe.

The command: built into /sma-execute-phase; resumption via /sma-resume-work.

10. V2 pillar: the report

What it does. One panel shows the whole picture: active sessions, predictions and their per-area accuracy, reflex firings, terminal collisions, memory health. The state of the system is visible, not assumed.

The command: /sma-progress in the chat; the data is assembled from journals and git.

Everything is driven by words in the chat. The full set:

12. What happens on its own, without you

  • At the start of every conversation: the window registers itself and loads the memory core.
  • Before every file edit: a check whether the file is claimed by another terminal.
  • Before every action: a check whether it matches the firing condition of a known reflex.
  • Every collision is written to the journal, so the real collision rate can be measured.
  • Dead entries from closed windows are cleaned up automatically.

The core principle: the system never blocks work. It warns and records, but does not stop you. If the memory or coordination layer itself breaks, work continues as before, just without the hints. Hard denies are reserved for security checks only.

13. Try it right now: 10 minutes

  1. Install: npx sma-framework init, then open a Claude Code session and type /sma-start.
  2. Type into the chat: pnpm sma status. You will see how many windows are working and what they are busy with.
  3. Open a second terminal. In the first one, claim some folder via pnpm sma claim.
  4. In the second one, ask the assistant to edit a file from that folder: a warning with the first terminal's name appears before the edit.
  5. In both windows at once: pnpm sma next-slot migration. The numbers come out different, no race.
