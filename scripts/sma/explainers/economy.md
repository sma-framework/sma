# Economy meters

Budgets from your own measured history, a versioned corpus token meter, and a self-cost meter, with every saving paired with a quality guard.

## en
SMA meters its own cost before it ever enforces a budget. Three surfaces:
- `memory stats` prices the memory corpus with a VERSIONED estimator (`ESTIMATOR_VERSION` is stamped in every report): the MEMORY.md core load, each note, each INDEX file, and the top-N heaviest. The numbers are an approximation, never billing truth, and they reproduce run-to-run.
- `spend self-cost` publishes the framework's OWN static per-session overhead: the SMA:RULES block, the emitted corpus block, and the MEMORY.md core. What it does not count (variable per-turn hook output) is named honestly.
- `spend lane open|close|reap|report|derive` carries a per-lane budget for the fix / quick / batch / build lanes. A budget derives ONLY from your own closed-run percentiles (p75). A lane with fewer than five closed clean runs gets no budget and stays report-only, so a soft signal never fires off a guess. On agent-driven terminals every CLI call is its own process, so `close` pairs with the newest open run — this terminal's first, else any terminal's, with the cross-terminal pairing named honestly; `reap` closes stranded opens as abandoned, permanently excluded from budget derivation.

An overrun is a scored calibration miss in the `sma.economy` domain plus an auto-drafted lesson. Runs a parallel terminal contaminated are excluded from derivation and never score a miss. Compress is deliberately deferred until `memory stats` shows measured pain. The ratchet law: a saving that costs quality is a miss, not a win.

The economy ladder (9.4-07) makes "write less" a checkable promise. Before writing code, walk the rungs in order: does this need to exist? already in this codebase? stdlib? native platform feature? installed dependency? one line? Only then the minimum that works. A plan CLAIMS its footprint in frontmatter (files, new files, ~loc, new deps, tolerance %); the grill standing challenge «which ladder rung?» (`grill <plan> --standing`) blocks a plan that carries no claim. At verify time `reverify --footprint <plan>` compares the claim against `git diff --numstat` actuals over the plan's commits, and an overrun beyond tolerance is a scored `sma.economy` miss. The floor is fixed: security, data-loss handling, trust-boundary validation, and accessibility are never on the chopping block.

The ship lanes (9.4-08) apply the same law to shipping. `ship-lane check` decides the ritual: a small reviewed delta (<= 5 commits, no migrations, no foreign push-claim) is eligible for `/sma-quick-ship`; anything else refuses with «this is a full /sma-ship». The quick lane cuts BATCH SIZE and babysitting, never a gate — the full test suite, fresh-types tsc, the security guard, and the V1.N tag all run identically in both lanes. `ship-lane changelog` drafts a deterministic conventional-commit changelog the full lane shares; `ship-lane record` logs each lane outcome so a quick lane that reds CI auto-tightens. CI and Railway are watched in the background so the founder is freed at push time.

## ru
SMA сначала измеряет собственную стоимость и только потом вводит бюджет. Три поверхности:
- `memory stats` оценивает корпус памяти ВЕРСИОНИРОВАННЫМ оценщиком (`ESTIMATOR_VERSION` проставлен в каждом отчёте): ядро MEMORY.md, каждую заметку, каждый файл INDEX и самые тяжёлые из них. Числа приблизительные, это не биллинговая правда, и они воспроизводимы от запуска к запуску.
- `spend self-cost` публикует СОБСТВЕННЫЕ статические накладные расходы фреймворка за сессию: блок SMA:RULES, вставленный блок корпуса и ядро MEMORY.md. То, что не учитывается (переменный вывод хуков за ход), названо честно.
- `spend lane open|close|reap|report|derive` несёт бюджет по полосам fix / quick / batch / build. Бюджет выводится ТОЛЬКО из перцентилей Ваших собственных закрытых прогонов (p75). Полоса, где меньше пяти чистых закрытых прогонов, бюджета не получает и остаётся только для отчёта, поэтому мягкий сигнал не срабатывает по догадке. На агентных терминалах каждый вызов CLI — отдельный процесс, поэтому `close` спаривается с новейшим открытым прогоном: сначала своего терминала, иначе любого, и межтерминальная пара называется честно; `reap` закрывает брошенные открытия как abandoned — они навсегда исключены из вывода бюджетов.

Перерасход это зачтённый промах калибровки в области `sma.economy` плюс автоматический черновик урока. Прогоны, загрязнённые параллельным терминалом, исключаются из вывода и промах не засчитывают. Сжатие корпуса намеренно отложено, пока `memory stats` не покажет измеренную боль. Закон храповика: экономия ценой качества это промах, а не победа.

Лестница экономии (9.4-07) превращает «пишите меньше» в проверяемое обещание. Перед тем как писать код, пройдите ступени по порядку: нужно ли этому вообще существовать? уже есть в этой кодовой базе? стандартная библиотека? возможность платформы? установленная зависимость? одна строка? И только потом минимум, который работает. План ЗАЯВЛЯЕТ свой объём в frontmatter (files, new_files, ~loc, new_deps, tolerance %); постоянный вызов grill «which ladder rung?» (`grill <plan> --standing`) не пропускает план без заявки. На проверке `reverify --footprint <plan>` сравнивает заявку с фактом `git diff --numstat` по коммитам плана, и перерасход сверх допуска это зачтённый промах `sma.economy`. Пол зафиксирован: безопасность, защита от потери данных, проверка на границе доверия и доступность никогда не идут под нож.

Полосы отправки (9.4-08) применяют тот же закон к пушу. `ship-lane check` выбирает ритуал: маленькая проверенная дельта (не более 5 коммитов, без миграций, без чужого push-claim) годится для `/sma-quick-ship`; всё остальное получает отказ «this is a full /sma-ship». Быстрая полоса сокращает РАЗМЕР ПАКЕТА и ручное ожидание, но не гейт: полный набор тестов, свежие типы с tsc, страж безопасности и тег V1.N выполняются в обеих полосах одинаково. `ship-lane changelog` готовит детерминированный черновик по префиксам, которым пользуется и полная полоса; `ship-lane record` пишет исход каждой полосы, поэтому быстрая полоса, покрасневшая в CI, сама сужается. CI и Railway отслеживаются в фоне, чтобы освободить основателя в момент пуша.
