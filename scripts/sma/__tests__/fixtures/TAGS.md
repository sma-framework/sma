# TAGS — закрытый фасетный словарь памяти (R1, D-49-12)

> **Это реестр авторитетных тегов (authority control, B3).** Тег, которого нет в
> этом файле, — это ОШИБКА линтера (49-08). Новый тег появляется только через
> правку этого файла в ТОМ ЖЕ коммите, что и первое использование. Так словарь
> остаётся закрытым при параллельной работе нескольких терминалов.
>
> **Параллельные добавления разрешаются как обычный git merge** (SPEC edge:
> concurrency R1): два терминала добавляют по строке — merge сливает обе, линтер
> перепроверяет реестр после merge. Реестр — это обычный коммитируемый файл, не
> база данных, не счётчик.
>
> **Фасеты (B1, Ranganathan / Z39.19):** заметка получает по одному тегу из
> `area` и из `kind`, плюс необязательный открытый `phase:NN`. Поиск = пересечение
> фасетов (например «tech + decision»). `area` — это ТОТ ЖЕ фасет, что в
> `.planning/BACKLOG.md` (crm/content/tech/governance/os), расширенный явными
> кластерами корпуса.
>
> **Синонимы (B2, ANSI/NISO Z39.19 USE/UF):** каждый канонический тег может нести
> `· aliases: a, b` — это UF-формы (Used For). Загрузчик и линтер приводят alias к
> канону (USE) перед сравнением. Так убирается дрейф синонимов (whatsapp/360dialog
> → messaging).
>
> **Грамматика строки (машиночитаемая, строгая):** под заголовком `## <facet>`,
> одна строка на тег:
> `- <tag> — <описание одной строкой> · aliases: <a>, <b>`
> Часть `· aliases:` необязательна. Разделитель ` — ` между тегом и описанием и
> разделитель ` · aliases: ` структурны — их потребляет парсер (loadTagsRegistry).

## area

- crm — заявки, клиенты, воронка, доска проектов, внутренние экраны /crm.
- content — публичный контент, SEO-страницы, блог-генератор, тексты для СНГ.
- tech — инфраструктура, сборка, типы, миграции, общая инженерия платформы.
- governance — правила, решения, DoD, аудит, комплаенс, источники истины.
- os — операционный фреймворк: бэклог, трекер, карта операций, ритм задач. · aliases: framework, internal-framework
- payload — Payload CMS: коллекции, хуки, Local API, доступ, поля. · aliases: cms
- railway — хостинг Railway: деплой, переменные, сборка, лимиты. · aliases: hosting, deploy
- messaging — каналы связи: WhatsApp, Telegram, VK, email, входящие. · aliases: whatsapp, 360dialog, wa, telegram, vk, comm, inbox
- memory — система памяти: заметки, индекс, теги, гигиена, SMA. · aliases: sma, notes
- security — безопасность: RBAC, вебхуки, секреты, XSS, регресс-гард. · aliases: sec, rbac
- phi — медданные: шифрование, маршрутизация LLM, приватность, GDPR. · aliases: privacy, gdpr
- seo — поисковая видимость: Яндекс, Google, схемы, GEO/AI-search. · aliases: geo
- design — визуальный дизайн, бренд, UI-токены, источник истины дизайна. · aliases: ui, brand
- workflow — рабочий процесс GSD: планирование, исполнение, проверка, фазы. · aliases: gsd, process
- release — версии и релизы: тег V1.N, changelog, гейт пуша. · aliases: version, push
- finance — расчёты, счета, взаиморасчёты по клиенту, отчёты. · aliases: billing
- agents — ИИ-агенты воронки и оркестратора, промпты, командный центр. · aliases: ai, orchestrator
- testing — тесты, фикстуры, vitest, tsc, гейты качества. · aliases: tests, ci

## kind

- procedural-rule — правило «как делать»: устойчивая инструкция, применяемая всегда. · aliases: rule, feedback-rule
- decision — зафиксированное решение (D-XX) с обоснованием и датой. · aliases: decision-record
- episodic — эпизод: что произошло в конкретной сессии, инцидент, исход работы. · aliases: episode, log
- status — текущее состояние: что живо, что готово, что за флагом, что блокирует. · aliases: state
- reference — справка: адреса, версии, ключи-факты, таблицы соответствий. · aliases: ref, lookup
- bug-lesson — урок из бага: заметка feedback_* со структурой **Why:** + **How to apply:** (D-49-15). · aliases: lesson, gotcha

## phase

- Открытый фасет: `phase:NN` — необязательный тег, привязывающий заметку к номеру
  фазы (например `phase:49`). Значение свободной формы `phase:<число>`; отдельные
  теги здесь не перечисляются.
