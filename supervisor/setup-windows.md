# SMA V5 — запуск ночного работника на Windows (обвязка Планировщика заданий)

Это windows-двойник `setup-macos.md`: тот же тонкий слой супервизора (D-9.5-02a), но
под Планировщик заданий Windows вместо launchd. Ядро демона host-agnostic; здесь только
привязка к ОС.

Порядок ниже проверен пилотом BL-94 на рабочем ПК основателя (без прав администратора,
без docker, без клиента psql). Шаги, которые выполняет основатель, написаны простым
языком. Инженерные детали идут ниже отдельными блоками.

> Важно про безопасность. Задача Планировщика поставляется ВЫКЛЮЧЕННОЙ и пилотом не
> регистрируется. Регистрацию и включение выполняет ТОЛЬКО основатель, вручную, после
> того как смоук ниже прошёл. Демон никогда не обращается к origin: у него нет пути к
> глаголу публикации. Одобренная работа возвращается тем, что основатель добавляет этот
> ПК как git-remote и забирает локальную ветку сам.

## 1. Что нужно заранее (prerequisites)

- **Node.js 18.17 или новее** и **git** в PATH. Демон использует только встроенные
  модули Node плюс `pg-boss`, поэтому проблема этого проекта с Node 24 и `@next/env`
  на демон НЕ распространяется (это чистый node без Next.js).
- **Локальный Postgres очереди на :5433.** Песочница `~/pg-sandbox` (embedded-postgres,
  PG18) уже настроена на этом ПК. Запуск: `cd ~/pg-sandbox && node start.mjs` (демон
  сам поднимается в фоне; повторный запуск безвреден). Здесь не нужны ни права
  администратора, ни docker, ни установленный сервер Postgres.
- **Зависимости демона.** `pgboss-backend.mjs` подключает `pg-boss` и `pg` лениво.
  Эти пакеты сейчас НЕ объявлены в `package.json` продукта (находка для волны
  развёртывания / плана 9.5-10). Пилот ставит их машинно-локально в `node_modules`
  (в git не попадает). Совместимая версия: `pg-boss@11` (у v12 нет default-export,
  который ждёт код; у v10 нет `getQueueStats`).

  ```powershell
  cd <SMA_HOME>
  pnpm add pg-boss@11 pg    # либо: npm install --no-save pg-boss@11 pg
  ```

## 2. Конфигурация машины (machine-local, в git НЕ попадает)

Файл `~/.sma-daemon/config.json` создаётся при первой загрузке демона со свежим
64-символьным токеном фронта. Создать его заранее одной командой:

```powershell
cd <SMA_HOME>
node --input-type=module -e "import('./daemon/src/config.mjs').then(m=>m.loadConfig())"
```

Затем задать в нём:

- `queueUrl` = `postgres://postgres:postgres@localhost:5433/sma_queue` — выделенная
  база очереди на :5433. НИКОГДА не база `postgres` и НИКОГДА не база Railway.
- `bind` = `127.0.0.1` (0.0.0.0 — только явное согласие, D-9.5-05).
- `workers` — для этого ПК один честный локальный работник:

  ```json
  { "id": "local-1", "lane": "prod", "provider": "claude",
    "account": { "name": "local-1", "configDir": "C:\\Users\\<you>\\.sma-accounts\\local-1",
                 "oauthTokenEnv": "SMA_LOCAL_1_TOKEN", "spendLogsDir": "C:\\Users\\<you>\\.sma-accounts\\local-1\\spend" },
    "enabled": true }
  ```

> Про режим 0600. На win32 `chmod` — задокументированный no-op; демон пытается его
> и молча игнорирует неудачу. Эквивалент 0600 на Windows обеспечивает ACL профиля
> пользователя: файл лежит в `~/.sma-daemon` под вашей учётной записью и по умолчанию
> читается только владельцем. Файл никогда не коммитится и никогда не печатается
> обёрткой/смоуком.

> Про реальную учётку. Привязка headless-аккаунта (`claude setup-token` -> токен в
> переменной окружения `SMA_LOCAL_1_TOKEN`) — часть НАСТОЯЩЕГО ночного прогона, а не
> смоука. Смоук проверяет контур очереди синтетической echo-задачей без трат LLM.

## 3. Приёмочный прогон — смоук контура (ЭТО и есть проверка пилота)

Сначала поднять песочницу Postgres (если ещё не запущена), затем запустить смоук:

```powershell
cd ~/pg-sandbox; node start.mjs      # поднять :5433 (безвредно, если уже запущен)
cd <SMA_HOME>
node supervisor/live-smoke-windows.mjs
```

Смоук доказывает на РЕАЛЬНОМ долговременном состоянии цепочку:
очередь принимает -> работник забирает из pg-boss на :5433 -> появляется квитанция
(receiptRef) -> результат виден в GET /api/state с токеном. Выход 0 = зелёно. Он также
проверяет отрицательную аутентификацию: GET /api/state без токена -> 401.

## 4. Регистрация и включение задачи — ТОЛЬКО ОСНОВАТЕЛЬ

Пилот эти команды НЕ выполняет. Их запускает основатель вручную, когда решит, что
ночной работник должен включиться. Перед импортом заменить `<SMA_HOME>` в
`sma-daemon-windows.task.xml` на абсолютный путь клона SMA.

```powershell
# выполняет только основатель
schtasks /Create /TN SMA-Daemon /XML supervisor\sma-daemon-windows.task.xml
schtasks /Change /TN SMA-Daemon /ENABLE
```

Задача поставляется с `<Enabled>false</Enabled>`, поэтому даже случайный импорт не
запустит работника. Снять её позже: `schtasks /Delete /TN SMA-Daemon /F`.

## 5. Дальнейшее (follow-up настоящего прогона)

- **Настоящий ночной прогон size:S бэклог-задачи** запускает основатель ПОСЛЕ включения
  задачи. Тогда демон реально забирает готовую задачу из бэклога, разворачивает
  worktree, спавнит работника и проходит reverify-гейт (единственная дверь в
  «выполнено», D-9.5-04a).
- **Отчёт в Telegram и экономика спенд-леджера** (остальная часть строки BL-94) едут
  на этом настоящем прогоне, а не на синтетическом смоуке. Смоук намеренно дешёвый:
  он доказывает контур, а не наполняет ночную статистику.

### Находки (READ-ONLY, для запаркованного плана 9.5-10 / волны развёртывания — НЕ патчились)

1. **tickDeps без `buildArgs`** (предсказано планом). `main.mjs` не передаёт `buildArgs` в тик, поэтому настоящая не-'built' задача не может дойти до спавна из продакшн composition-root. На этот смоук не влияет (дверь preflight пропускает спавн).
2. **`ledger` передан в тик как СТРОКА `ledgerDir`** (предсказано планом). Значит `ledger.recordAttempt` внутри `loop.mjs` (completeTask/failTask) — молчаливый no-op (гард `typeof === 'function'` его пропускает). НО строку attempt-ledger пишет САМ pg-boss-адаптер в своём `complete()` (ledgerDir проброшен в `createPgBossQueue`), поэтому леджер ВСЁ РАВНО наполняется — через путь адаптера, а не тика. Итог для деплоя: ветка леджера в тике мёртвая как проложена; единственная запись идёт из адаптера (что для одной записи корректно).
3. **`createDaemon().start()` не вызывает `adapter.start()`.** Возвращаемый `start()` поднимает только фронт и тик; соединение pg-boss не открывается. Смоук (и настоящий деплой) должен стартовать адаптер сам. Пункт для волны развёртывания.
4. **`pgboss-backend.mjs start()` ссылается на dead-letter очередь `sma.task.dead`, но не создаёт её.** Под pg-boss v11 `createQueue(lane, {deadLetter})` валидирует цель и падает с «Queue sma.task.dead does not exist» -> свежий бут крашится. Деплой (или сам backend) должен создать dead-letter очередь до lane-очередей. В пилоте её провиженит обвязка.
5. **`pg-boss` и `pg` не объявлены в `package.json` продукта**, хотя `pgboss-backend.mjs` подключает их лениво. Ставятся машинно-локально (в git не попадают). Совместимая версия — `pg-boss@11`: у v12 нет default-export, который ждёт `const { default: PgBoss } = await import('pg-boss')`; у v10 нет `getQueueStats`; у v11 нет `touch` (но `touch` в этот контур не входит — дверь preflight не спавнит и не стримит). Волне развёртывания стоит объявить рантайм-зависимости и закрепить совместимый мажор pg-boss.
